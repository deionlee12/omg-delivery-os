How to Read This Report

Disposition Framework

Every configuration dimension is assigned one of five dispositions. These tell the migration team what to do with each element — not just whether it is compatible, but whether it should be carried forward at all. DC didn’t break. It evolved into a configuration model that Cloud has structurally replaced. This assessment maps that gap precisely.

MigrateDirect Cloud equivalent exists. Carry forward with standard migration tooling. May require minor reconfiguration.
Native CloudThe DC configuration was a workaround for missing functionality. Cloud has this natively. Remove the workaround and configure the native feature instead.
RethinkThe DC approach exists in Cloud but is not best practice. Migration is possible but redesign before or during migration is the right move.
RemoveNo Cloud equivalent, zero active usage, or the feature has been superseded. Do not migrate. Archive or decommission before cutover.
ConditionalWhether to migrate depends on license tier, active usage, or stakeholder decision. Requires investigation before disposition is confirmed.
Part 1

Jira Software DC → Cloud

Jira Software is the most configuration-rich of the three products. The most critical DC-to-Cloud differences involve Groovy-based customisations (ScriptRunner, Behaviours), the project hierarchy model, and the shift from server-side post-functions to Cloud Automation. Every architectural decision below flows from these three shifts.

RETHINK Workflows ▶
DC Approach
Workflow schemes with Groovy post-functions, validator scripts, and condition scripts via ScriptRunner
Cloud Equivalent
Shared workflow schemes; post-functions replaced by Automation rules; conditions via role validators
Recommendation

Migrate workflow status/transition structure. Remove all Groovy post-functions and reimplement as Automation rules. Remove ScriptRunner conditions — replace with role validators or Automation.

RETHINK ScriptRunner Scripts ▶
DC Approach
Groovy scripts for workflow post-functions, validators, conditions, field defaults, and event listeners
Cloud Equivalent
No Groovy in Cloud. Automation rules cover ~70% of use cases. Complex logic requires Forge apps.
Recommendation

Audit all scripts. Categorise as: (a) replaceable by Automation, (b) requires Forge app, (c) obsolete. Do not migrate scripts — rebuild as Automation rules or Forge apps.

RETHINK Behaviours (Field Logic) ▶
DC Approach
Behaviours app: show/hide fields, set required dynamically, cascade field values based on other field input
Cloud Equivalent
No direct equivalent. Partial coverage via field configurations + Automation rules for cascading. Forge UI for complex dynamic behaviour.
Recommendation

Catalogue all Behaviours rules. Simple required/hidden rules migrate to field configurations. Cascade logic requires Automation. Complex UI behaviour requires a Forge app — budget development time accordingly.

MIGRATE Issue Types ▶
DC Approach
Issue type schemes with custom types; Advanced Roadmaps adds Portfolio Epics and Initiatives
Cloud Equivalent
Same structure in Cloud. Advanced Roadmaps (Premium) supports Initiatives. Level names may differ.
Recommendation

Migrate issue type schemes directly. If using Advanced Roadmaps hierarchy, confirm target org has Jira Premium. Level names may need updating for Cloud AR naming conventions.

MIGRATE Custom Fields ▶
DC Approach
Global custom fields; field configuration schemes control required/hidden per issue type and project
Cloud Equivalent
Same model in Cloud. Context-scoped fields available. Global fields are an anti-pattern at scale.
Recommendation

Migrate with field context audit. Re-scope global fields to project or issue-type context. Remove fields with zero usage in 180 days. De-duplicate fields with identical name and type.

MIGRATE Permission Schemes ▶
DC Approach
Permission schemes with all grant types — user-level grants are unsupported best-practice-wise
Cloud Equivalent
Same model with group/role grants. Migrate schemes; convert user grants to groups.
Recommendation

Migrate permission schemes. Convert all user-level grants to group membership before migration. Remove Application Role grants — not applicable in Cloud.

MIGRATE Notification Schemes ▶
DC Approach
Notification schemes with user, group, role, and email recipients; event-driven
Cloud Equivalent
Same model in Cloud. Email recipient notifications should be replaced with group/role.
Recommendation

Migrate notification schemes. Audit for individual email and user recipients — convert to group/role. Reduce event subscriptions to high-signal events only.

MIGRATE Screen Schemes ▶
DC Approach
Screen and screen schemes; issue type screen schemes
Cloud Equivalent
Identical in Cloud
Recommendation

Migrate directly. Audit for screens with 20+ fields — Cloud best practice is to keep screens focused. Remove unused screens.

DC Approach
Limited native automation (auto-assign, auto-close). Most automation via ScriptRunner.
Cloud Equivalent
Jira Cloud Automation is a first-class feature with 50+ trigger types, conditions, and actions. Far more capable than DC built-in.
Recommendation

Do not migrate DC automation rules. Rebuild all ScriptRunner-based automation logic as Cloud Automation rules. Cloud Automation is the replacement for ScriptRunner listeners and post-functions.

MIGRATE Project Roles ▶
DC Approach
Project roles with user and group actors; global role definitions
Cloud Equivalent
Same in Cloud; group actors preferred
Recommendation

Migrate role definitions. Convert user actors to group membership. Remove project-specific user actor assignments.

Cloud Equivalent

Migrate directly. Archive released versions before migration. Audit component leads — orphaned components need owners assigned.

Jira Software boards: Scrum and Kanban; filter-based boards; Rapid Views (legacy)
Cloud Equivalent
Same board types in Cloud. Rapid Views (legacy Jira boards) do not exist in Cloud.
Recommendation

Migrate Scrum and Kanban boards. Rapid View configurations must be rebuilt as standard boards. Audit board filters — project key references will change. Rebuild column-to-status mappings explicitly.

CONDITIONAL Advanced Roadmaps ▶
DC Approach

Cloud Equivalent
Advanced Roadmaps is native in Jira Cloud Premium. Data model and hierarchy levels differ from DC Portfolio.

RETHINK Apps / Plugins ▶
DC Approach
DC Marketplace apps (ScriptRunner, Tempo, Zephyr, Elements, etc.)
Cloud Equivalent
Cloud Marketplace has different apps. Not all DC apps have Cloud equivalents. Some have rewritten Cloud versions with different feature sets.
Recommendation

Audit every installed DC app. Confirm Cloud equivalent exists and supports required features. Identify apps with no Cloud version — these require native Cloud or Forge alternatives. Remove unused apps before migration.

RETHINK User Directory ▶
DC Approach
Local users, LDAP sync, Crowd integration, manual user management
Cloud Equivalent
Atlassian Cloud identity: Atlassian Access (SAML SSO, SCIM provisioning, policy enforcement). No local users.
Recommendation

Do not migrate local users directly. Set up Atlassian Access with your IdP (Okta, Azure AD, etc.) before migration. Map DC user accounts to cloud-managed identities. Local user accounts must be converted — there is no bulk import path for passwords.

MIGRATE Saved Filters & Dashboards ▶
DC Approach
User-owned and shared filters with JQL; dashboards with gadgets
Cloud Equivalent
Same in Cloud. Some gadget types do not exist in Cloud (Agile gadgets replaced by boards).
Recommendation

Migrate filters and dashboards. Update JQL to use Cloud project keys. Remove gadgets with no Cloud equivalent. Audit shared filter permissions — group sharing must use Cloud groups.

MIGRATE Issue Security Schemes ▶
DC Approach
Security levels with user, group, role, and reporter grants
Cloud Equivalent
Same in Cloud
Recommendation

Migrate directly. Audit for user-level security grants — convert to groups.

RETHINK Email / Incoming Mail ▶
DC Approach
Incoming mail handlers: create issues, add comments from email
Cloud Equivalent
Cloud has email integration through a simplified handler. Complex routing logic requires automation.
Recommendation

Audit incoming mail handler logic. Simple create-issue-from-email migrates to Cloud email integration. Complex routing rules must be rebuilt as Automation rules with email triggers.

Jira Service Management DC → Cloud

JSM has significant architectural differences between DC and Cloud. The Cloud version has substantially more native capability — particularly around virtual agents, conversational ticketing, and asset management. Many DC configurations built around email-driven workflows and manual SLA management can be replaced with Cloud-native automation. The most time-intensive dimension is Assets / Insight — plan it separately from the main migration.

2.1 Architecture Differences — JSM

ConceptData CenterCloud
Virtual agent / chatbotNot available natively; required third-party appNative Virtual Agent in JSM Cloud (Premium/Enterprise)
Asset managementAssets (formerly Insight) as a DC pluginAssets is native in JSM Cloud; schema and import differ from DC Insight
Customer portalService desk portal; limited brandingHelp Center with richer branding, topic groups, and portal customisation
Request typesRequest type groups; forms via plugin or portalNative forms with conditional logic (no plugin required)
SLA definitionsSLA configurations with calendarsSame structure; Cloud adds SLA countdown widgets and native reporting
ApprovalsBasic approval step in workflowsEnhanced approvals: multi-level approvals, approval groups
Email requestsEmail handler with manual routingEmail integration with Automation for routing; dedicated email address per queue
Incident managementRequires add-on (OpsGenie integration via plugin)Native incident management in JSM Cloud Premium; OpsGenie integrated natively

2.2 Configuration Dimension Analysis — JSM

RETHINK Request Types ▶
DC Approach
Request type groups in portal; fields mapped from issue fields; limited conditional display
Cloud Equivalent
Cloud native forms with conditional field logic, rich text, attachment fields, date pickers natively
Recommendation

Do not migrate request types directly. Redesign using Cloud native forms — conditional logic, field descriptions, and help text are now native. Remove request types with zero submissions in 180 days.

MIGRATE SLA Configurations ▶
DC Approach
SLA definitions with start/stop/pause conditions and working calendars
Cloud Equivalent
Same structure in Cloud. Working calendars migrate. Reporting is richer in Cloud.
Recommendation

Migrate SLA definitions. Validate working calendar timezone settings — Cloud uses UTC-based definitions. Re-test SLA breach alerts post-migration.

MIGRATE Working Calendars ▶
DC Approach
Custom working hours and holiday calendars per service desk
Cloud Equivalent
Same in Cloud; timezone representation differs
Recommendation

Migrate working calendars. Audit timezone configuration — DC stores in server timezone, Cloud uses explicit timezone. Re-validate SLA calculations post-migration.

MIGRATE Queues ▶
DC Approach
Agent queues with JQL filters; manual queue ordering; queue groups
Cloud Equivalent
Same in Cloud; personal/starred queues are Cloud-only additions
Recommendation

Migrate queue definitions. Update JQL to use Cloud project keys. Add starred queues for high-priority agents — this is a Cloud-only best practice with no DC equivalent.

RETHINK Customer Portal / Help Center ▶
DC Approach
Service desk portal with project-level branding; portal groups
Cloud Equivalent
Help Center with global branding, topic groups, announcement banners, and featured portals
Recommendation

Do not migrate portal config directly. Redesign Help Center layout using Cloud's richer branding options. Migrate topic groups. Remove portal groups with zero customers.

RETHINK Email Integration ▶
DC Approach
Email handlers: create requests, add comments, resolve via email keywords
Cloud Equivalent
Dedicated email addresses per service project; Cloud Automation for routing and keyword processing
Recommendation

Migrate email address assignments. Rebuild keyword-based routing and auto-resolution as Cloud Automation rules. Complex multi-project routing requires Automation with conditional logic.

NATIVE Automation Rules (JSM) ▶
DC Approach
ScriptRunner listeners for SLA breach alerts, escalation routing, and approval triggers
Cloud Equivalent
JSM Cloud Automation with JSM-specific triggers: SLA breach, request type, queue entry, approval events
Recommendation

Remove all ScriptRunner-based JSM automation. Rebuild as Cloud Automation with JSM triggers. SLA breach automation is native in Cloud — no scripting required.

RETHINK Approvals ▶
DC Approach
Single-level workflow approval step; approver set via field or role
Cloud Equivalent
Cloud approvals support multi-level, group-based, and dynamic approver resolution. Native approval email/notification.
Recommendation

Migrate basic approval steps. Redesign multi-level approval flows using Cloud native capabilities — do not replicate complex approval chains using workflow hacks that were required in DC.

RETHINK Assets / Insight (CMDB) ▶
DC Approach
Insight (DC plugin): object schemas, attribute types, object relationships, import rules, references in fields
Cloud Equivalent
Assets native in JSM Cloud. Schema structure is similar but import configuration and API differ. DC data migrated via CSV export/import.
Recommendation

Plan Assets migration separately. Export Insight object data as CSV. Re-create object schemas in Cloud Assets before importing. Object reference fields in requests must be re-linked after import. This is the most time-intensive dimension of the JSM migration.

NATIVE Incident Management ▶
DC Approach
Incident tracking via issue type; OpsGenie integration via marketplace plugin; manual escalation
Cloud Equivalent
Native incident management in JSM Cloud Premium: incident timelines, stakeholder updates, on-call schedules, OpsGenie native integration
Recommendation

Migrate incident issue type and workflow. Connect OpsGenie (if in use) via native Cloud integration — remove the DC plugin. Enable incident management features in JSM Cloud Premium. Redesign escalation flows using native on-call and alert routing.

MIGRATE Knowledge Base ▶
DC Approach
Confluence integration for KB articles; manual linking to portal
Cloud Equivalent
Native Confluence Cloud integration; Knowledge Base auto-suggestion in portal; article feedback metrics
Recommendation

Migrate KB Confluence integration configuration. Cloud KB has enhanced article suggestion and deflection metrics — configure these after migration to reduce ticket volume.

MIGRATE Permission Schemes (JSM) ▶
DC Approach
Service project permission schemes; customer role; agent role
Cloud Equivalent
Same in Cloud; customer portal access controlled separately from Jira permissions
Recommendation

Migrate permission schemes. Audit customer access settings — Cloud separates portal customers from licensed users more clearly than DC.

RETHINK Reports & Dashboards (JSM) ▶
DC Approach
Built-in SLA reports; custom JQL dashboard gadgets; third-party reporting apps
Cloud Equivalent
Cloud native JSM reports; Atlassian Analytics (Premium/Enterprise); third-party Cloud apps
Recommendation

Migrate JQL-based dashboard gadgets. Rebuild SLA reporting using Cloud native JSM reports. Evaluate Atlassian Analytics for advanced reporting — this replaces most third-party reporting apps.

Part 3

Confluence DC → Cloud

Confluence Cloud has diverged significantly from DC/Server in its editor, macro set, and integration model. The most impactful differences are: the Fabric (Cloud) editor vs. the legacy editor, the macro library (many DC macros are deprecated or replaced), and the introduction of Confluence Databases and Smart Links as native features. Deprecated macros are a hard migration blocker — they must be resolved before migration tooling runs.

Macros~120 DC macros including many server-side macrosSubset of macros; many DC macros deprecated; Cloud-native features replace several macro categories

MIGRATE Space Permissions ▶
DC Approach
Space-level permissions: view, add pages, add comments, add attachments, delete, admin — per user and group
Cloud Equivalent
Same permission model in Cloud; anonymous access works differently (public links preferred)
Recommendation

Migrate space permissions. Convert user-level grants to groups. Audit anonymous access spaces — Cloud uses shareable public links for external access rather than anonymous Confluence access.

RETHINK Page Templates ▶
DC Approach
Space-level and global page templates; Blueprint templates
Cloud Equivalent
Same in Cloud with additional Fabric editor-native templates. DC templates using deprecated macros will not render correctly.
Recommendation

Audit all templates. Templates using deprecated macros must be rebuilt in the Fabric editor. Remove templates with zero usage in 12 months. Migrate templates that use only supported macros.

RETHINK Macros ▶
DC Approach

Cloud Equivalent

Run the Atlassian Cloud Migration Assistant — it identifies all macro usage and flags deprecated macros before migration. Do not migrate pages with unresolved deprecated macros without a remediation plan.

Many DC macros built on server-side execution: Scaffold, Metadata, SQL for Confluence, RefinedWiki macros, etc.
Cloud Equivalent
No equivalent — these macros do not run in Cloud
Recommendation

Identify all pages using deprecated macros via the Cloud Migration Assistant. For each: determine if Cloud-native feature replaces it (Databases for table macros, Smart Links for embed macros). Pages with unresolvable macros must be manually reviewed before migration.

RETHINK ScriptRunner for Confluence ▶
DC Approach
Groovy scripts for: auto-labelling pages, bulk content operations, custom macros, event listeners
Cloud Equivalent
Confluence Cloud Automation (native) replaces ~60% of use cases. Forge apps for custom macros.
Recommendation

Audit all Confluence ScriptRunner scripts. Rebuild auto-labelling and bulk operations as Cloud Automation rules. Custom macro scripts require Forge apps. Remove scripts with zero execution history.

MIGRATE Space Structure ▶
DC Approach
Spaces with page hierarchies; personal spaces; archived spaces; space categories
Cloud Equivalent
Same in Cloud; space archiving is more prominent; team spaces are a Cloud concept
Recommendation

Migrate space structure. Archive inactive spaces before migration rather than migrating then archiving. Remove personal spaces for inactive users. Confirm space key collisions do not exist (space keys are instance-unique in Cloud).

Confluence Databases native in Cloud: structured data with views, filters, and rollup fields

Identify all pages using structured table macros, embedded spreadsheets, or database-style plugins. Rebuild as Confluence Databases post-migration. This is one of the most impactful Cloud improvements for operations and project tracking use cases.

Cloud Equivalent
Smart Links native (100+ integrations); Confluence Whiteboards; Atlassian native diagram tools

Page-level restrictions: view and edit restrictions per user and group

Confluence Cloud Automation: page creation triggers, scheduled rules, label-based actions, archiving rules

Native integrations in Cloud: Slack, Teams, GitHub, Figma, and more. Atlassian Rovo for AI-assisted content.

NATIVE Analytics ▶

Native page analytics in Cloud. Atlassian Analytics (Premium) for workspace-level reporting.

MIGRATE Global Templates & Blueprints ▶
DC Approach
Global templates available to all spaces; Blueprint templates with wizard-style creation
Cloud Equivalent
Same in Cloud; Cloud Blueprint library includes new Fabric-native templates
Recommendation

RETHINK User / Group Permissions ▶
DC Approach
Local groups, LDAP groups, Confluence-specific groups
Cloud Equivalent
Atlassian Access org-level groups; IdP-synced groups via SCIM
Recommendation

Map DC Confluence groups to Cloud org-level groups before migration. User migration requires Atlassian Access setup with IdP. Do not attempt to migrate user credentials — Cloud users authenticate via IdP or Atlassian account.

Part 5

Migration Readiness & Remediation Plan

Migration tasks are sequenced across four phases. Phase 1 (DC clean-up) must complete before migration tooling runs. Phase 2 (Cloud build-out) runs in parallel with Phase 1 where possible. The order is not flexible — each phase creates the preconditions the next requires. Attempting these out of sequence compounds risk rather than reducing it.

Environments that consolidate without stabilising dependencies first consistently produce more post-cutover incidents than the preparation work would have taken. Items within a phase that carry no inter-dependencies can run in parallel to compress the timeline. Nothing else should be accelerated.

Phase 1DC Instance Clean-Up — All Three ProductsWeeks 1–4

P0 M01 Audit and categorise all ScriptRunner scripts (Jira + Confluence)

Scripts are the single biggest migration risk across all three products. Uncategorised scripts become dead code in Cloud or block migration tooling. This is the first task on Day 1.

  1. Export full script inventory: Groovy scripts, workflow conditions, post-functions, validators, listeners, and Confluence scripts
  2. For each script: classify as (a) replaceable by Cloud Automation, (b) requires Forge app, or (c) obsolete/unused
  3. Mark obsolete scripts for removal — do not migrate any script to Cloud
  4. Create a backlog of Automation rebuilds and Forge app requirements from classifications (a) and (b)
  5. Prioritise Automation rebuilds that unblock workflow migration (post-functions and validators first)

P0 M02 Resolve deprecated Confluence macros

  1. Run the Atlassian Cloud Migration Assistant to generate a full deprecated macro report

  3. Prioritise pages with the highest view counts first — these are the most user-visible

  4. Map DC account usernames to Cloud account email addresses (these may differ)

Part 6

Cloud Best Practice Summary

These best practices apply across all three products in the Cloud environment. They represent the “do it right in Cloud” principles that should guide configuration decisions during and after migration. These are not suggestions — they are the governance baseline the Alkami Technology Cloud environment should be built to from Day 1.

6.1 Cross-Product Principles

  • Identity first: Configure Atlassian Access and IdP integration before any migration work begins. All subsequent user migration depends on it.
  • No Groovy: Cloud has no Groovy runtime. Every DC script must be categorised as Automation, Forge, or remove — before migration, not after.
  • Automation over configuration: Cloud Automation is the primary mechanism for workflow logic, notifications, and cross-product events. Invest in Automation design before go-live.
  • Group-based access: All permission grants, notification recipients, and role actors must use groups or roles. No individual user references — they are a maintenance liability and break with IdP sync.
  • Remove before migrating: Everything unused, deprecated, or superseded should be removed from DC before migration tooling runs. Migrating technical debt to Cloud amplifies it.
  • Native over plugins: Before installing any Cloud app, verify the use case is not already covered natively — Assets, Databases, Smart Links, Automation, Incident Management, and Analytics are all native in Cloud.

6.2 Jira Software Cloud Best Practices

  • Use a single shared workflow per issue type across projects — divergent workflows at scale are the most common cause of reporting inconsistency in Cloud
  • Scope custom fields to project or issue-type context — global fields are an anti-pattern; they appear in all project field pickers across the org
  • Create per-team filter-based boards in consolidated projects — one board per team, not one board per project
  • Use Atlassian Analytics (Premium) for cross-project reporting instead of JQL-based dashboard gadgets for complex metrics
  • Standard hierarchy only (Epic → Story/Bug/Task → Sub-task) unless Advanced Roadmaps Premium is licensed

6.3 Jira Service Management Cloud Best Practices

  • Use Cloud native forms with conditional logic for all request types — no plugin required
  • Define SLA calendars in explicit timezones (UTC-based) — DC timezone inheritance does not apply in Cloud
  • Enable article suggestion in Help Center Knowledge Base — Cloud deflection metrics show which articles are reducing ticket volume
  • Use starred/personal queues for agent prioritisation — this is a Cloud-only capability with no DC equivalent
  • Configure Assets object schemas before importing data — schema changes after import are painful
  • Use multi-level native approvals rather than workflow hacks for approval chains

6.4 Confluence Cloud Best Practices

  • Run the Cloud Migration Assistant before every migration attempt — macro compatibility and user mapping issues surface here, not during migration
  • Use Confluence Databases for all structured data that previously lived in table macros or embedded spreadsheets
  • Use Smart Links for all external content references — paste URL, Smart Links renders the preview
  • Archive spaces with zero activity before migration — archiving in DC is free; archiving 500 unused Cloud spaces is a project in itself
  • Rebuild templates in the Fabric editor before migration — templates with deprecated macros will not render correctly post-migration
  • Enable native page analytics immediately post-migration — engagement data from Day 1 informs content quality decisions
What success looks like

After migration, Alkami Technology operates from a single, governed Cloud environment — built for the capabilities Cloud was designed to deliver.

No more Groovy
All workflow logic lives in Cloud Automation — visible, testable, and maintainable without developer intervention for routine changes.
Identity is governed
Atlassian Access + IdP means user provisioning and deprovisioning happens in one place. No orphaned accounts. No permission drift.
Plugins replaced by native capability
Insight → Assets. ScriptRunner → Automation. Rapid Views → Standard boards. Each replacement is simpler, more capable, and lower maintenance.
Content that actually renders
Deprecated macros resolved before cutover. Fabric editor for all content. Confluence Databases for structured data. Smart Links for everything else.

DC didn’t fail. It reached the natural end of a configuration model that Cloud has structurally replaced. This migration is not a repair — it’s a deliberate transition to a platform built for how Alkami Technology works today.
