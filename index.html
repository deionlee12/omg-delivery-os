01

Purpose

OMG Delivery OS is the operating system for running a high-quality Atlassian consulting practice — from the first client conversation through delivery, expansion, and renewal.

It gives the team a single source of truth for how work gets done, how clients are managed, and how the business performs — without adding bureaucracy or slowing down delivery.

The goal is simple: less heroics, more repeatability. Every client should get the same quality of experience. Every engagement should produce data that makes the next one better.

02

Operating Principles

🎯

Clarity over completeness

A clear, working system beats a perfect system that no one uses. Start with the minimum that creates real visibility and build from there.

🔁

Every engagement produces data

Actuals feed estimates. Retros feed process. Every project should make the next one faster and more profitable.

👤

Ownership is non-negotiable

Every stage, every client, every deliverable has one named owner. Shared accountability is no accountability.

🚫

No silent scope creep

Out-of-scope work is logged, assessed, and approved before it's done. Always. Change orders protect both parties.

📊

Margin is a lagging indicator

By the time margin suffers, something upstream broke. We track hours, scope, and rhythm weekly — not at close.

⚡

Speed through structure

Structure doesn't slow delivery down — ambiguity does. Clear gates, clear owners, and clear outputs compress timelines.

Leadership View

📊 What Leadership Sees Weekly

Pipeline health — deals by stage, estimated close

Delivery status — active projects, current phase, any blockers

Hours vs estimate — per project, per workstream

Margin outlook — flagged if any project exceeds 10% variance

Escalations — anything requiring principal involvement

✅ What Delivery Leads Own

Execution — kickoff through close, no gaps

Schedule + scope — updated weekly, escalated promptly

Client communication — status, decisions, risks

Change control — nothing absorbed silently

Margin visibility — hours in Tempo from day one

📈 What We Measure

Est. vs actual hours — by stage

Gross margin — per project

Cycle time — SOW → launch

Scope change events — frequency and reason

Customer wait time — schedule days lost to client delays

Rework hours — signals estimate or scope issues

Current Build Priorities

1

Delivery model — live on Arhaus Live

9-stage delivery model with stage gates, Tempo tracking, and weekly margin visibility in production on the Arhaus engagement. Data from this project will validate the model and inform fixed-fee pricing on future deals.

2

HubSpot pipeline → delivery handoff Now

Clean deal stages in HubSpot that mirror the customer lifecycle. Automatic task creation when a deal reaches "SOW Signed" to trigger the delivery lead kickoff checklist.

3

Estimation library Now

A structured log of estimated vs actual hours by engagement type and phase. Starts with Arhaus actuals. Used to improve pre-sales and pre-SOW estimates and build toward fixed-fee confidence.

4

Weekly leadership dashboard Next

A single Confluence page updated weekly: pipeline status, active project health, hours vs estimate, and any active escalations. One place, five minutes, full picture.

5

Proposal + SOW template library Next

Standardized templates in PandaDoc for the four most common engagement types: Jira implementation, JSM/ITSM, migration, and assessment-only. Reduces proposal time and improves scope consistency.

6

Customer success playbook Later

Post-launch check-in schedule, expansion conversation triggers, and NPS process. Builds on delivery model retro data and ensures every client has a named success path after go-live.
